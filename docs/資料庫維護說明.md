### **方式 2.1: 補休 IMPORT baseline 精準修復**
當歷史資料曾多次匯入補休基準時，可能留下多筆 `referenceType = 'IMPORT'` 的 baseline。修復工具會保留最新一筆 IMPORT baseline，刪除較舊 baseline，並依最新 baseline 重新計算 `compLeaveBalance`。

```bash
# 只檢查，不寫入資料
npm run repair:comp-leave-imports

# 指定單一員工檢查
npm run repair:comp-leave-imports -- --employeeId=123

# 對指定 snapshot 或正式資料庫做 dry-run
npm run repair:comp-leave-imports -- --database=/absolute/path/to/snapshot.db

# 以 JSON 取得 dry-run 結果，方便留存或比對
npm run repair:comp-leave-imports -- --database=/absolute/path/to/snapshot.db --json

# 單一員工 apply 不需要額外 confirm
npm run repair:comp-leave-imports -- --database=/absolute/path/to/snapshot.db --employeeId=123 --apply

# 全庫 apply 需要額外確認字串
npm run repair:comp-leave-imports -- --database=/absolute/path/to/snapshot.db --apply --confirm=REPAIR_ALL_IMPORT_BASELINES
```

**重要安全規則：**
- 不加 `--apply` 時預設為 dry-run。
- 加上 `--json` 時，CLI 會只輸出 dry-run JSON，不再混入說明文字。
- 對全庫使用 `--apply` 時，必須額外加上 `--confirm=REPAIR_ALL_IMPORT_BASELINES`；若有 `--employeeId=<id>` 則不需要。
- 若未提供 `DATABASE_URL` 與 `-- --database=...`，fallback 會指向 `file:./prisma/dev.db`。
- `-- --database=...` 可傳 SQLite 路徑或 `file:` URL，CLI 會把來源標示為「命令列參數」。
- 工作區內的 `prisma/prod.db` 目前是 0-byte 空檔，不應視為可用的 production snapshot。
- 真正修復前，應先對 production DB 或 snapshot 做 dry-run，再決定是否加 `--apply`。

## 🎯 核心維護操作

### **1. 狀態檢查**
```bash
# 檢查內容
- 資料庫檔案完整性
- WAL 模式狀態  
- 緩存配置
- 資料表統計
- 回應時間測試
```

### **2. 性能優化**
```sql
-- 自動執行的優化項目
PRAGMA journal_mode = WAL;        -- WAL 模式
PRAGMA cache_size = -10240;       -- 10MB 緩存
PRAGMA foreign_keys = ON;         -- 外鍵約束
PRAGMA synchronous = NORMAL;      -- 同步模式
PRAGMA temp_store = MEMORY;       -- 記憶體暫存
REINDEX;                         -- 重建索引
ANALYZE;                         -- 更新統計
```

### **3. 備份管理**
```bash
# 備份類型
- 每日自動備份 (壓縮)
- 手動完整備份
- 緊急備份 (故障前)

# 備份位置
/backups/daily/     - 每日備份
/backups/weekly/    - 每週備份  
/backups/emergency/ - 緊急備份
```

### **4. 資料清理**
```sql
-- 清理項目
DELETE FROM SystemLog WHERE created_at < datetime('now', '-3 months');
DELETE FROM TempData WHERE created_at < datetime('now', '-1 year');  
DELETE FROM PasswordResetToken WHERE expires_at < datetime('now');
VACUUM;  -- 釋放空間
```

## 🚨 故障排除

### **常見問題**

#### **1. 資料庫鎖定**
```bash
問題: database is locked
解決: echo "BEGIN IMMEDIATE; ROLLBACK;" | sqlite3 prisma/dev.db
```

#### **2. 性能下降**
```bash
解決步驟:
1. 執行性能優化
2. 清理緩存
3. 重建索引 
4. 檢查磁碟空間
```

#### **3. 檔案損壞**
```bash
檢查: sqlite3 prisma/dev.db "PRAGMA integrity_check;"
修復: sqlite3 prisma/dev.db "REINDEX;"
恢復: 從最新備份恢復
```

### **緊急恢復**
```bash
# 1. 立即停止服務
pm2 stop attendance-system

# 2. 備份當前狀態  
cp prisma/dev.db prisma/dev.db.emergency

# 3. 從備份恢復
cp /backups/latest/attendance.db prisma/dev.db

# 4. 重啟服務
pm2 start attendance-system
```
